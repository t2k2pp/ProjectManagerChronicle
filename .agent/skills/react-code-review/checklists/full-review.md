# フルレビュー チェックリスト

詳細なコードレビュー用（30分以上）

---

## 1. コード品質・可読性

### 命名規則
- [ ] 変数名: camelCase を使用
- [ ] コンポーネント名: PascalCase を使用
- [ ] ブール値: is/are/should/has などのプレフィックス
- [ ] イベントハンドラ: handle/on プレフィックス
- [ ] 名前が目的を明確に表現している

### フォーマット・スタイル
- [ ] インデント・空白が一貫している
- [ ] 長すぎる行がない（80-120文字推奨）
- [ ] 適切な空行で論理的に区切られている

### DRY 原則
- [ ] 重複コードがカスタムフックやユーティリティに抽出されている
- [ ] 類似コンポーネントが共通化されている
- [ ] 定数が適切にまとめられている

### 複雑性
- [ ] 関数が単一責任を持つ
- [ ] ネストが深すぎない（3段階以下推奨）
- [ ] 条件分岐が複雑すぎない
- [ ] 早期リターンが活用されている

### コメント・ドキュメント
- [ ] 複雑なロジックに説明がある（Why を説明）
- [ ] TODO/FIXME が適切に管理されている
- [ ] JSDoc が必要な箇所に記述されている

---

## 2. React ベストプラクティス

### コンポーネント設計
- [ ] コンポーネントサイズが適切（200-300行以下）
- [ ] JSX が50行以下
- [ ] 関心の分離ができている（ロジック/UI）
- [ ] 適切な粒度で分割されている
- [ ] 再利用可能な設計になっている

### Hooks の使用
- [ ] Hooks がトップレベルでのみ呼ばれている
- [ ] useEffect の依存配列が正確
- [ ] useEffect のクリーンアップが実装されている
- [ ] カスタムフックへの抽出が適切
- [ ] useCallback/useMemo が必要な箇所で使用されている

### Props と State
- [ ] Props が最小限で意味がある
- [ ] Props の型定義が明確
- [ ] Props が直接変更されていない
- [ ] State が適切なスコープで管理されている
- [ ] 派生値が State ではなく計算で得られている
- [ ] Props のデストラクチャリングが活用されている

### リスト・繰り返し
- [ ] key に一意で安定した値が使用されている
- [ ] 配列の index が key に使用されていない
- [ ] 大きなリストに仮想化が検討されている

### 条件付きレンダリング
- [ ] 条件が明確で読みやすい
- [ ] && 演算子での falsy 値の扱いが正しい
- [ ] 三項演算子が複雑すぎない

---

## 3. TypeScript 型安全性

### 型定義
- [ ] `any` 型が回避されている
- [ ] `unknown` で代替できる箇所は unknown を使用
- [ ] Props/State に明示的な型がある
- [ ] 関数の引数・戻り値に型がある
- [ ] イベントハンドラの型が正しい

### 型の活用
- [ ] ユーティリティ型を活用（Pick, Omit, Partial, Required）
- [ ] ユニオン型・Enum で制限された値を表現
- [ ] ジェネリクスが適切に使用されている
- [ ] 型の再利用・共通化ができている

### 型の整理
- [ ] 機能/ドメイン単位で整理されている
- [ ] interface と type の使い分けが一貫している
- [ ] readonly/as const が適切に使用されている
