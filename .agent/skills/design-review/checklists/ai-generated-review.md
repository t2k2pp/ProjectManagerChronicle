# AI生成設計 専用レビュー チェックリスト

生成AIが作成した設計書の特有の問題を検出するためのチェックリスト

---

## 1. 省略・不完全パターンの検出

### 列挙の不完全さ
- [ ] 「など」「等」「といった」で終わる列挙がないか
- [ ] 「以下に示す」と言いつつ一部しか示されていないか
- [ ] 「主な～」「代表的な～」で始まる不完全なリストがないか
- [ ] 「この他にも」で追加が示唆されつつ具体化されていないか

### 先送り表現
- [ ] 「詳細は別途定義」「後続工程で決定」がないか
- [ ] 「TBD」「TODO」「検討中」「未定」がないか
- [ ] 「追って連絡」「別途調整」がないか
- [ ] 「必要に応じて追加」が放置されていないか

### 未完了の参照
- [ ] 存在しない章・節への参照がないか
- [ ] 「後述する」と書かれて実際に後述されていないか
- [ ] 「別紙参照」「添付参照」で添付がないか
- [ ] 「関連ドキュメント参照」の具体的なドキュメント名がないか

---

## 2. 抽象化・曖昧さの問題

### 定性的すぎる表現
- [ ] 「高速に」「効率的に」「適切に」など具体的な数値がないか
- [ ] 「大量の」「少量の」がどれくらいか定義されていないか
- [ ] 「リアルタイム」の具体的な応答時間がないか
- [ ] 「十分な」「適度な」がなにを基準にしているか不明でないか

### 具体性の欠如
- [ ] エラーコードが「ERR001」のような仮番号のままでないか
- [ ] エラーメッセージが「エラーが発生しました」のような汎用的なものでないか
- [ ] データ型が「文字列」「数値」のみで桁数・範囲がないか
- [ ] 状態が「処理中」「完了」のみで中間状態がないか

### 判断基準の不明確さ
- [ ] 「状況に応じて」「場合によって」の判断基準がないか
- [ ] 「システム管理者が判断」の判断基準がないか
- [ ] 条件分岐の「その他」がどういうケースか不明でないか
- [ ] 優先順位が「高・中・低」だけで具体的な順序付けがないか

---

## 3. コピペ・テンプレートパターン

### 同一フレーズの繰り返し
- [ ] 説明文が複数箇所でほぼ同一でないか
- [ ] 異なる機能なのに同じ処理詳細が書かれていないか
- [ ] 「〇〇を実施する」が具体的な手順なく繰り返されていないか
- [ ] エラー処理が全箇所で同じ記述（「エラー処理を実施」等）でないか

### テンプレート的記述
- [ ] 「[機能名]を実行する」のような穴埋め形式が残っていないか
- [ ] 「ユーザーが○○する」ばかりで例外ケースがないか
- [ ] すべての項目が同じ文章構造になっていないか
- [ ] 具体例がなにもない抽象的な記述のみでないか

---

## 4. 人間視点の欠落

### 運用観点
- [ ] 障害発生時の対応手順がないか
- [ ] 手動でのリカバリ手順がないか
- [ ] 運用者が見るべき監視ポイントがないか
- [ ] データパッチ・緊急対応の手順がないか
- [ ] 問い合わせ時に必要な情報（ログ、ID等）が明記されていないか

### ユーザー観点
- [ ] ユーザーが理解できるエラーメッセージになっているか
- [ ] 操作ミス時のリカバリ方法があるか
- [ ] ヘルプ・ガイダンスが考慮されているか
- [ ] 待ち時間が長い処理のフィードバックがあるか

### 開発者観点
- [ ] デバッグ時に必要な情報（ログレベル、トレースID等）があるか
- [ ] テスト時の前提条件・テストデータが考慮されているか
- [ ] 設定変更時の影響範囲が明記されているか
- [ ] デプロイ手順・ロールバック手順が考慮されているか

---

## 5. エッジケース・異常系の欠如

### 境界値
- [ ] 最大値・最小値の処理が定義されているか
- [ ] 空（ゼロ件、空文字、null）の処理が定義されているか
- [ ] 上限を超えた場合の処理が定義されているか
- [ ] 文字数・桁数オーバーの処理が定義されているか

### 異常系
- [ ] ネットワークエラー・タイムアウト時の処理があるか
- [ ] 外部システム障害時のフォールバックがあるか
- [ ] セッション切れ・認証エラー時の処理があるか
- [ ] 排他制御での競合時の処理があるか

### 特殊ケース
- [ ] 初回起動時・初期データがない場合の処理があるか
- [ ] 移行期間中の新旧混在ケースが考慮されているか
- [ ] 時間帯依存（営業時間外、メンテナンス時間等）が考慮されているか
- [ ] 祝日・年末年始などの特殊日の処理があるか

---

## 6. 論理的矛盾・不整合

### 条件の矛盾
- [ ] AならばBと書かれた後、Aなのに not B となるケースがないか
- [ ] 排他的であるべき条件が重複していないか
- [ ] 条件分岐が網羅的でなく、どこにも該当しないケースがないか
- [ ] 循環参照・循環依存がないか

### データの不整合
- [ ] 同じデータが異なる属性で定義されていないか
- [ ] 親子関係・依存関係が矛盾していないか
- [ ] 必須項目がオプションとして扱われているケースがないか
- [ ] 計算式・導出値の定義が矛盾していないか

---

## 7. 楽観的仮定

### 正常系への偏り
- [ ] 「成功した場合」ばかりで失敗ケースがないか
- [ ] 「ユーザーが正しく入力した場合」ばかりでないか
- [ ] 外部システムが常に正常動作する前提でないか
- [ ] データが常に整合している前提でないか

### リソースの楽観視
- [ ] メモリ・ストレージが無限にある前提でないか
- [ ] ネットワーク帯域が常に十分な前提でないか
- [ ] 処理時間がゼロに近い前提でないか
- [ ] 同時アクセスが少ない前提でないか

---

## チェック方法

### キーワード検索
以下のキーワードを検索して問題箇所を特定：
```
など|等|といった|以下に|主な|代表的|この他|詳細は別途|後続|TBD|TODO|検討中|未定|追って|必要に応じて|適切に|状況に応じて|場合によって|高速|効率的|リアルタイム|十分|適度
```

### パターン確認
1. 同じ段落が繰り返されていないか目視確認
2. すべての条件分岐で「その他」「それ以外」の処理を確認
3. エラー処理の記述が具体的か確認
4. 数値で表すべき項目（時間、件数、サイズ）に具体的な値があるか確認

### 人間の視点で質問
- 「これを見た開発者は何をすればいいかわかるか？」
- 「障害が起きたとき、運用者はどう対応すればいいか書いてあるか？」
- 「ユーザーがエラーを見たとき、どうすればいいかわかるか？」
- 「テストするとき、どういうデータ・条件で試せばいいかわかるか？」
