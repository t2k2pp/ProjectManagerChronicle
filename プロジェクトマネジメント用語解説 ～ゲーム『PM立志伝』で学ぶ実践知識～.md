# プロジェクトマネジメント用語解説 ～ゲーム『PM立志伝』で学ぶ実践知識～

## はじめに：ゲームで学ぶ、プロジェクトマネジメントの世界

プロジェクトマネジメント（PM）の世界へようこそ。この分野には**「ステークホルダー」**や**「WBS」**、**「EVM」**といった、一見すると難解に聞こえる専門用語が数多く存在します。しかし、これらはプロジェクトを成功に導くための強力な「武器」であり、その本質は決して複雑なものではありません。

この資料では、架空のシミュレーションゲーム『PM立志伝：プロジェクト・クロニクル』を題材に、プロジェクトマネジメントの重要用語を解説します。ゲームの具体的な機能やイベントと関連付けることで、初心者の方でも各用語の役割と重要性を直感的に理解し、楽しみながら学べるように構成されています。

--------------------------------------------------------------------------------

## 1. プロジェクトの全体像を掴むための基本用語

このセクションでは、プロジェクト全体を管理し、その「健康状態」を把握する上で基礎となる3つの重要な概念を紹介します。これらは、いわばプロジェクトという船の航海計器のようなものです。

### 1.1 PMBOK (プロジェクトマネジメント知識体系)

**PMBOK**（ピンボック）とは、プロジェクトマネジメントにおける世界標準の知識体系であり、先人たちの成功と失敗から編み出された「成功のための羅針盤」のようなものです。

#### ゲーム『PM立志伝』での役割

このゲームでは、運や勘に頼ったプレイでは決して成功できません。勝利の鍵は、PMBOKの考え方に沿った論理的な判断を下すことです。

このゲームの勝利条件は、運ではなく**「PMBOK（プロジェクトマネジメント知識体系）に準拠した正しい判断」**です。

ゲーム内のイベントや選択肢は、すべてPMBOKのフレームワークに基づいて設計されています。プレイヤーは成功体験を通じて、その考え方の有効性を自然と体感し、論理的な意思決定のプロセスを学ぶことができます。

### 1.2 ステークホルダー (Stakeholder)

**ステークホルダー**とは、プロジェクトの成否に影響を与えたり、影響を受けたりするすべての人々を指します。顧客、上司、部下、協力会社など、その範囲は多岐にわたります。彼らとの良好な関係を築き、期待を調整することが、プロジェクトマネージャー（PM）の最重要任務の一つです。

#### ゲーム『PM立志伝』での役割

ゲームには、性格や能力値が細かく設定された多様なステークホルダーが登場します。例えば、「売上至上主義の上司」はリスクの高い案件を強要してきたり、「新技術に抵抗するベテラン職人」はチームの方針に反発したりします。あるいは、常に笑顔で同意してくれる理想的な顧客に見えながら、プロジェクト終盤に「やっぱりイメージと違う」と笑顔でちゃぶ台返しをする`『微笑みの仕様粉砕者』西園寺 麗子`のような手強い相手も登場します。

プレイヤーは、こうしたキャラクターたちの立場や性格を理解し、それぞれに合わせたコミュニケーション（交渉や説得）を取らなければなりません。この仕組みを通じて、ステークホルダー・マネジメントの重要性と難しさを実践的に学ぶことができます。

### 1.3 QCD (品質・コスト・納期)

**QCD**とは、プロジェクトの成功を測るための3つの主要な指標です。

- **Q (Quality):** 品質
- **C (Cost):** コスト（予算）
- **D (Delivery):** 納期

これら3つは密接に関連しており、多くの場合「何かを優先すれば、何かが犠牲になる」という**トレードオフ**の関係にあります。

#### ゲーム『PM立志伝』での役割

ゲームでは、プロジェクトが完了するたびに、この`QCD（品質・コスト・納期）の達成度`に基づいて最終的なスコア（利益や会社の信頼度）が決定されます。

例えば、プレイヤーが「品質を上げる」ために優秀なエンジニアを追加で投入すると、コストが増加し、場合によっては納期も遅延します。逆に、納期を守るためにテスト工程を省略すれば、品質が低下し、リリース後にバグが多発して信頼を失うかもしれません。例えば、「売上至上主義の上司」というステークホルダーは、常に品質(Q)を犠牲にしてでもコスト(C)と納期(D)を優先する判断を迫ってくるでしょう。誰が何を重視しているかを見極めることが、QCDバランスの鍵となります。プレイヤーは常にこの3つのバランスを意識した、現実的な判断を迫られることになります。

これらの基本用語は、プロジェクトの健康状態を把握するための計器のようなものです。次に、プロジェクトを具体的に動かすための「計画」に関する用語を見ていきましょう。

## 2. 成功の礎を築く「計画」の用語

プロジェクトの成否は、実行前の「計画」の質で8割が決まるとも言われます。このセクションでは、成功の礎を築く計画フェーズで使われる重要な用語を解説します。

### 2.1 WBS (作業分解構成図)

**WBS** (Work Breakdown Structure) とは、巨大で漠然としたプロジェクト全体を、管理可能なレベルまで細かく分解し、階層構造で整理する手法です。これにより、作業の抜け漏れを防ぎ、関係者全員がプロジェクトの全体像を正確に把握できるようになります。

#### ゲーム『PM立志伝』での役割

ゲームの計画フェーズでは、WBSの作成が「**パズル形式でタスクを分解する**」ミニゲームとして表現されています。プレイヤーは大きなタスクブロックを、より小さなタスクブロックへと論理的に分割していきます。

このパズルを解く過程で、プレイヤーは自然と「どのように作業を分解すれば、依存関係が明確になり、管理しやすくなるか」を考えるようになります。これにより、WBS作成の思考プロセスを体験的に学ぶことができます。

### 2.2 クリティカルパス

**クリティカルパス**とは、WBSで分解されたタスクの中で、プロジェクト全体の納期を決定する「最も時間のかかる一連の作業経路」のことです。このパス上にあるタスクが1日でも遅れると、プロジェクト全体の納期が1日遅れることになります。PMはこのパスを最優先で管理する必要があります。

#### ゲーム『PM立志伝』での役割

WBS作成パズルの中では、**「クリティカルパス（最長経路）を特定し、そこにリソースを集中させる」**ことが正解の行動として設定されています。ゲームはこのクリティカルパスのタスクに、最も優秀なメンバーを割り当てることで高評価が得られるように設計されています。これによりプレイヤーは、PMとして最も重要な『リソースを集中投下すべき一点』を見抜く訓練を積むことになるのです。

### 2.3 リスクマネジメント

**リスクマネジメント**とは、「将来起こりうるかもしれない問題（リスク）」を事前に洗い出し、その影響を最小限に抑えるための対策を立てておく活動のことです。覚えておいてください。プロジェクトマネジメントとは、問題が起きてから対処することではなく、問題が起きないように先手を打つことです。リスク登録簿は、そのための未来予測ツールなのです。

#### ゲーム『PM立志伝』での役割

ゲームの計画フェーズには、「リスク登録簿」という機能があります。ここに考えられるリスクを記載しておくと、実行フェーズで実際にトラブルが発生した際のダメージを大幅に軽減できます。

これらはランダムに見えますが、計画段階での「リスク登録簿」への記載有無でダメージが変わります。リスク対策済みなら「予備費（コンティンジェンシー予備）」を使ってノーダメージで切り抜けられます。

この仕組みを通じて、プレイヤーは「事前の備え」がプロジェクトの安定にどれほど貢献するかを痛感します。行き当たりばったりの進行ではなく、常に先を見越して管理することの重要性を学ぶことができるのです。

しっかりとした計画が立てば、いよいよプロジェクトの実行です。しかし、計画通りに進まないのが常。次に、実行中に発生する問題に対処するための用語を学びましょう。

## 3. 荒波を乗りこなす「実行・監視」の用語

プロジェクトは生き物です。計画通りに進むことは稀で、日々状況は変化します。このセクションでは、プロジェクトの実行中に状況を正確に把握し、適切に対応するために不可欠な用語を解説します。

### 3.1 EVM (アーンド・バリュー・マネジメント)

**EVM**（Earned Value Management）とは、プロジェクトの健全性を「予定通りか？」と「予算通りか？」という２つのシンプルな問いに、客観的な数値で答えるための手法です。これにより「進捗率90%」といった曖昧な報告ではなく、「計画よりもお金を使いすぎている」「進捗は進んでいるが、実は計画より遅れている」といった客観的な事実を明らかにします。

#### ゲーム『PM立志伝』での役割

ゲームのメイン画面上部には、常に`EVMグラフ`が表示されています。このグラフは「PV（計画値） vs EV（出来高）」と「AC（実コスト）」の関係を示しており、**プロジェクトの健康状態が一目でわかる**ようになっています。

このグラフは、プロジェクトが静かに「炎上」へと向かっていることを示す早期警戒システムです。この兆候を見逃すことは、手遅れを意味します。

### 3.2 スコープ・クリープ

**スコープ・クリープ**とは、プロジェクトの途中で、当初の計画（スコープ）に含まれていなかった仕様変更や追加要求が、なし崩し的に増えていく現象のことです。「ついでにこれもお願い」が重なり、気づいた時には予算も納期も破綻している、というプロジェクト炎上の主要な原因の一つです。

#### ゲーム『PM立志伝』での役割

ゲーム内の「交渉カードバトル」では、顧客からの追加要求は「攻撃」として扱われます。これに対し、プレイヤーは防御カードである`『スコープ・クリープ』`を使い、「それは契約範囲外です」と宣言して攻撃を無効化できます。

このカードバトルを通じて、プレイヤーは追加要求に対して毅然と断ることや、正式な変更手続きを要求することの重要性を、ゲームに勝利するという形で学ぶことができます。

### 3.3 トレードオフ

**トレードオフ**とは、「何かを得るために、別の何かを犠牲にしなければならない」という関係性のことです。プロジェクトマネジメントでは、前述の**QCD**（品質・コスト・納期）の間で頻繁に発生します。例えば、「納期を短縮する」ためには、「コストを増やす（人を増やす）」か「品質を下げる（テストを減らす）」といった決断が必要になります。

#### ゲーム『PM立志伝』での役割

交渉カードバトルには、`『トレードオフ』`という交渉カードが登場します。このカードの効果は以下の通りです。

『トレードオフ』: 「納期を優先するなら品質を落とします」と提案し、相手の攻撃力を下げる。

このカードは、無理な要求をされた際に、ただ「No」と断るのではなく、「Aを優先するならBは諦めていただく必要がありますが、いかがですか？」と代替案を提示して交渉するという、より高度なPMのコミュニケーションスキルを学ぶ機会を提供します。

## 4. 次の成功へ繋げる「終結」の用語

### 4.1 KPT (ケプト)

**KPT**（ケプト）とは、プロジェクトの「振り返り」を行うためのシンプルで強力なフレームワークの一つです。以下の3つの観点で議論し、経験を次に活かすことを目的とします。

- **Keep:** 良かったこと、今後も**続けるべきこと**
- **Problem:** 悪かったこと、課題として認識すべきこと
- **Try:** Problemを踏まえ、**次に挑戦すること**

#### ゲーム『PM立志伝』での役割

プロジェクトが完了すると、「振り返りフェーズ」が始まります。ここではKPTフレームワークが採用されており、プロジェクト中の成功や失敗から得られた経験値が「教訓」として会社の資産として保存されます。そして、この「教訓」は、**次回のプロジェクトで自動的にパラメータにボーナスがつく**という形で還元されます。

このシステムは、単にプロジェクトを成功・失敗で終わらせるのではなく、「失敗から学び、次の成功の糧にする」という、プロジェクトマネジメントにおける継続的な改善サイクルと思考法を、プレイヤーに自然と身につけさせる効果があります。

## まとめ

ここまで解説してきたPMBOK、ステークホルダー、WBS、EVMといった用語は、単なる知識ではありません。これらは、複雑で予測不可能なプロジェクトという荒波を乗りこなし、チームを成功に導くための実践的な「武器」です。

このガイドは『PM立志伝』の攻略本であると同時に、あなたのキャリアにおけるロードマップでもあります。ゲームで戦略を試し、現実世界で成果を出してください。

さあ、この世界で存分に失敗し、学び、そして伝説のPMを目指してください。あなたの挑戦が、現実世界での成功の礎となることを信じています。
