# PM立志伝：プロジェクト・クロニクル 設計仕様書

## 1. 基本情報
- **タイトル**: PM立志伝：プロジェクト・クロニクル
- **ジャンル**: プロジェクトマネジメント・シミュレーションRPG
- **コンセプト**:
  - **PMBOK（プロジェクトマネジメント知識体系）に基づく意思決定の重要性**: 運への依存を最小化し、適切な知識と状況判断がプロジェクトの成否を決定する設計とする。
  - **経営シミュレーションとRPG要素の統合**: 組織運営・人材育成の要素と、個人のキャリア形成・対人交渉の要素を融合する。
  - **実践的学習ツール**: ステークホルダーマネジメントやリスク管理など、PMの実務能力向上を主眼に置く。

---

## 2. ゲームシステム概要

### 勝利条件
適切なPM知識と論理的な状況判断に基づき、プロジェクトの目標（QCD：品質・コスト・納期）を達成すること。

### プロジェクト管理プロセス
1プロジェクトは以下の4フェーズで構成される。
1. **立ち上げ (Initiating)**: 案件選定、プロジェクト憲章の策定、ステークホルダーとの合意形成。
2. **計画 (Planning)**: チーム編成、WBS（作業分解構成図）の作成、クリティカルパスの特定、リスク登録簿の作成。
3. **実行・監視 (Executing & Monitoring)**: 週次サイクルの進行、課題管理およびトラブルシューティング。
4. **終結 (Closing)**: 検収・リリース、KPT法による振り返り、学習レポートの生成。

### ターン進行：週次PDCAサイクル
1ターンを1週間（月曜～金曜）とし、PDCAサイクルに基づいて進行する。
- **月曜（Plan）**: タスクアサイン、リソース配分、週間方針（品質優先/納期優先等）の策定。
- **火～木（Do/Check）**: 自動進行および監視。イベント発生時は具体的対応（Judgment）を選択・介入する。
- **金曜（Action）**: 週次実績評価（EVM確認）、次週に向けた対策立案。

### 12ヶ月カレンダーシステム
日本の会計年度（4月～3月）を採用し、時期に応じた外部環境の変化を実装する。
- **高負荷時期（繁忙期）**: GW、お盆、年末年始などの長期休暇前後は、稼働率低下や連絡遅延のリスクが増大する。
- **要員調整フェーズ**: 大型連休の2週間前に発生。メンバー各々の休暇取得方針に基づき、稼働確保のための交渉（特別報酬提示、説得、代休調整等）を行う。

## 3. キャラクター設計

### パラメータ体系：ダブル・ヘキサゴン・システム
全キャラクターは「プロジェクトスキル（表）」と「チーム運営能力（裏）」の2種、計12項目のパラメータ（各10段階評価）を持つ。

#### 【表：プロジェクトスキル（実務遂行能力）】
1. **設計 (Design)**: 仕様不備や手戻りの抑制能力。
2. **製造 (Develop)**: 実装速度および作業処理能力。
3. **評価 (Test)**: 品質保証能力および潜在バグの検出力。
4. **折衝 (Negotiation)**: 仕様変更要求や短納期要求に対する防御・交渉力。
5. **提案 (Propose)**: 課題解決のためのアイデア創出力および打開力。
6. **判断 (Judgment)**: 優先順位の策定、自律的な問題解決力、最適解の選択能力。

#### 【裏：チーム運営能力（組織貢献能力）】
1. **庶務 (Admin)**: 管理工数の削減および事務処理能力。
2. **幹事 (Organizer)**: チームの士気向上および結束力の強化。
3. **給仕 (Service)**: メンバーの体力維持および疲労回復支援。
4. **雑談 (Chat)**: メンバー間の対立解消および人間関係の円滑化（調整役）。
5. **美貌 (Charm)**: 周囲への能力向上補正（自己効力感の喚起）、および対外的な印象形成力。
6. **幸運 (Luck)**: 低確率で発生するポジティブなイベント（外部要因によるトラブル回避等）。

### 体力・回復特性システム
年齢層に基づき、「最大体力（連続稼働限界）」と「回復力（疲労回復速度）」の特性が定義される。

| タイプ | 年齢層モデル | 特徴 | 運用指針 |
| :--- | :--- | :--- | :--- |
| **瞬発型** | 20代 | 最大体力：低 / 回復力：高 | 短期集中型。高負荷な稼働後も短期間で回復可能。 |
| **持続型** | 30-40代 | 最大体力：高 / 回復力：低 | 長期耐久型。長期間の高負荷に耐えるが、枯渇時のリカバリに長時間を要する（蓄積疲労リスク）。 |
| **省エネ型** | 50代～ | 最大体力：低 / 回復力：低 | マネジメント特化型。「判断」スキル等を活用した司令塔としての運用を推奨。 |

### 休暇取得指針（Work-Life Trait）
裏パラメータとして、以下の行動特性を保持する。
- **休暇重視型**: 休暇取得を最優先する。休暇制限は離職リスクに直結する。
- **報酬重視型**: 金銭的インセンティブを重視。休日出勤手当等による稼働確保が可能。
- **計画重視型**: 早期の予定確定を重視。直前のスケジュール変更に対し強い抵抗を示す。
- **定時重視型**: ワークライフバランスを重視。所定労働時間内のパフォーマンスは高いが、時間外労働に適さない。

### 固有NPCと覚醒システム
- **二つ名開眼 (True Name Reveal)**: 信頼度等の特定条件を満たすことで、隠された真の能力（二つ名）が解放される。
- **潜在能力の顕在化**: ランダム生成された一般キャラクターにおいても、信頼度が最大化することで「実務特化」や「特定スキル特化」などの潜在能力が発現する場合がある。

## 4. プレイヤー（主人公）設定
ゲーム開始時に以下の設定を行い、初期ステータスおよび特性を決定する。

1. **年齢層の選択**:
   - **若年層**: 高い回復力を保持するが、スキルは未熟。現場密着型スタイル。
   - **中年層**: バランス型。標準的な能力構成。
   - **高年層**: 体力は低いが、高度なスキル（判断・折衝等）を保持。マネジメント特化型スタイル。

2. **PM適性診断**:
   - トラブル対応方針等に関する設問への回答に基づき、初期ボーナスおよびPMタイプを決定する。
   - **タイプ分類例**: 技術主導型、営業主導型、管理主導型、リーダーシップ特化型など。

## 5. 詳細ゲームシステム

### 交渉および対話システム
ステークホルダー（顧客、上層部、開発チーム）との利害対立を解決する。
- **交渉カード**: 「スコープ管理（防御）」「トレードオフ（代替案提示）」「心理的安全性確保（回復）」等のカードを使用する。
- **直接交渉（Free Input）**: 重要局面において使用可能。プレイヤーが入力した説得内容をAIが「論理性・配慮・実現可能性」の観点で評価し、結果に反映する。

### 世界環境の初期生成
シミュレーションの公平性を担保するため、開始時に以下の環境変数を固定生成する。
- **自社プロファイル**: 初期資金、信用格付け、組織風土。
- **人材ＤＢ**: 全社員のパラメータ、潜在特性、相関関係。
- **市場環境**: 競合他社および協力会社（BP）の特性、稼働状況。
- **案件パイプライン**: 年間の案件発生計画（リソース計画の基礎情報）。

### 学習振り返りレポート (AI Feedback)
ゲーム終了時、プレイログを解析し、AIが「プロジェクト完了報告書」形式でフィードバックを出力する。
- **行動特性分析**: プレイヤーの意思決定傾向の客観的評価。
- **重要局面の評価**: 岐路となった判断に対する妥当性の検証と解説。
- **用語解説**: ゲーム内で触れたPM用語の定義と実務的文脈の補足。
- **KPT分析**: Keep（継続すべき点）、Problem（課題）、Try（次回の改善点）の提示。

## 6. UIデザイン：PMコックピット

情報の視認性を最優先し、戦況把握に適したダッシュボード形式を採用する。

- **上部（Head-Up Display）**: カレンダー（季節変動・イベント予兆）、EVMメーター（SPI：スケジュール効率指数 / CPI：コスト効率指数）、残予算管理。
- **中央左（Office View）**: 座席表形式によるリソース状態表示。各種状態アニメーション、体力ゲージ（タイプ別識別）、思考状態（アラート等）の可視化。
- **中央右（Gantt Chart）**: ガントチャートによる進捗管理。計画と実績の対比、要員アサイン状況、リスク予兆の可視化。
- **下部（Control Deck）**: アサイン操作、全体方針指示、ログ参照などの操作系を集約。

## 7. 技術スタック
- **プラットフォーム**: Flutter (Mobile/Web) または Tauri (Desktop)
- **AI基盤**:
  - ローカルLLM (Ollama等) または 外部API (Gemini/OpenAI)
  - 用途：自然言語による入力判定、レポート生成、NPC会話生成
- **データ構造**:
  - JSON形式によるキャラクターデータの共通化。
  - `traits`（特性タグ）システムによる拡張性の確保。
